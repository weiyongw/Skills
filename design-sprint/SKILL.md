用户有模糊想法，想做新功能/模块但还没想清楚。通过结构化探索，帮用户从想法收敛到明确方案，产出完整设计参考文档。

## 核心原则

- **不猜，多问** — 没确认范围不出图，拿不准就让用户选
- **ASCII 先行** — 先对齐布局逻辑，确认后再投入 HTML
- **一次多方案** — 批量出 5-8 个 ASCII 方案，不要一个个试
- **全状态必须覆盖** — 正常态只是起点，异常态和边界情况必须穷举
- **决策落纸** — 每个确认的决策都写进需求总结

## 输出物

归档到 `设计/v{版本号}-{模块名}/`（版本号和模块名问用户确认）：

| 文件 | 用途 |
|------|------|
| `需求总结.md` | 背景、目标、功能范围、关键决策、技术约束 |
| `{模块名}-设计稿.html` | 主界面 HTML mockup |
| `{模块名}-全状态设计参考.html` | 所有状态、Toast、边界情况、交互规则表 |

## 工作流程

### 1. 需求收敛

用户说了模糊想法后，追问三件事：
- **痛点** — 遇到什么问题？为什么做？
- **核心场景** — 最典型的使用场景？
- **边界** — 这个版本做什么？不做什么？载体是什么？

整理成"做什么 / 不做什么 / 载体"的结构化共识。**没对齐边界不出图。**

### 2. 技术调研（按需）

涉及外部数据（配置文件、API、第三方系统）时必须先调研技术约束。纯 UI 功能跳过。

### 3. ASCII 批量探索

一次出 **5-8 个**不同思路的 ASCII 布局方案，每个附一句话说明核心思路。方案间差异要明显。

用户可以选一个、组合多个、或全部否掉补充新方向。

### 4. 确认设计风格

问用户三个问题：
1. **有无已有设计规范？** — 有则严格遵循 design tokens
2. **有无参考页面？** — 有则对齐外框样式
3. **载体形式？** — 已有应用需对齐导航结构

**没问就不默认使用任何设计规范。**

### 5. HTML 设计稿

基于选定方案 + 设计风格输出 HTML mockup，使用真实数据填充。让用户浏览器查看，根据反馈微调。

### 6. 全状态覆盖

产出全状态设计参考 HTML，必须包含以下四部分：

**页面状态**：正常态、加载态、空态、搜索无结果、错误态（按场景拆分）、部分可用

**交互反馈**：所有操作的成功/失败/警告 Toast 汇总，含具体文案、位置、时长规则

**边界情况**：长文本截断、大量数据滚动、其他场景特定情况

**交互行为规则表**（前端直接对照实现）：

| 触发 | 用户操作 | 系统行为 | 反馈 |
|------|---------|---------|------|
| 用户/系统/异常 | 具体操作 | 处理步骤 | Toast/状态变化 |

文档结构：标题版本日期 → 目录锚点 → 各状态 section → 交互规则表

### 7. 需求总结 + 归档

按模板（`templates/需求总结-模板.md`）生成需求总结，主动收集对话中所有决策写入"关键决策"。三个文件归档到约定目录。
